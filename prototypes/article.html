从零构建个人博客系统：
Node.js + EJS 实践全记录

📅 2025年12月20日 📖 12 分钟阅读 ✍️ 博主

历时三周，我用 Node.js、Express 和 EJS 从零搭建了这个博客系统。这篇文章记录了技术选型的思考、踩过的坑，以及最终让我满意的架构设计——不追求花哨，只求可维护。

为什么选择这套技术栈

在开始之前，我考虑过 Next.js、Gatsby、Hugo 等方案。但最终回到 Node.js + Express + EJS，原因很简单：我想真正理解每一行运行的代码，而不是被框架的魔法遮蔽。

"真正的理解是：你能从头实现它。" —— 某位不知名的程序员

EJS 是服务端渲染的模板引擎，语法接近原生 HTML，学习曲线极低。对于个人博客这种内容型产品，SSR 的 SEO 友好性远优于纯 SPA。

系统架构设计

整体分为三层：路由层（Express Router）、控制器层（Controller）和数据层（MySQL + 连接池）。没有引入 ORM，直接使用参数化查询——对于这种规模的项目，ORM 带来的复杂度远大于收益。

src/
├── routes/          # Express 路由定义
│   ├── index.js     # 前台路由（文章列表、详情、关于）
│   └── admin.js     # 后台路由（登录、管理、编辑）
├── controllers/     # 业务逻辑
│   ├── article.js
│   └── admin.js
├── views/           # EJS 模板
│   ├── layout/      # 公共布局（head, nav, footer）
│   ├── pages/       # 页面模板
│   └── partials/    # 可复用片段
└── app.js           # 入口

数据库设计

数据模型保持克制——只有两张核心表：articles 和 tags，通过中间表 article_tags 关联。

文章表核心字段

  • id：自增主键
  • slug：URL 友好的唯一标识（如 build-blog-with-nodejs）
  • title、summary、content：标题、摘要、正文（Markdown 存储）
  • status：draft / published
  • published_at、created_at、updated_at

索引策略

为 status + published_at 创建联合索引，覆盖最常见的查询模式（已发布文章按时间倒序）。

踩过的坑

最大的教训来自 SQL 注入防护。早期版本我直接拼接 SQL 字符串——这是非常危险的做法。后来全部改为参数化查询：

// ❌ 危险：直接拼接
const sql = `SELECT * FROM articles WHERE id = ${req.params.id}`;

// ✅ 安全：参数化查询
const sql = `SELECT * FROM articles WHERE id = ?`;
const [rows] = await pool.execute(sql, [req.params.id]);

部署到阿里云 SAE

SAE（Serverless App Engine）是阿里云的无服务器应用引擎，免运维、自动弹性，非常适合个人项目。通过环境变量注入数据库连接信息，代码与配置完全分离。

博客系统已上线运行，源码在 GitHub 上开源。如果你也想搭一个自己的博客，这套方案是个不错的起点——足够简单，足够可控。
